## RAG retrieval endpoints and evaluation

Summary:
- Added two read-only HTTP endpoints for RAG retrieval inspection and evaluation before agent integration.
- POST /rag/search: Ad-hoc query endpoint for manual Swagger testing. Accepts query, top_k, and optional destination_filter.
- GET /rag/eval: Fixed evaluation set with 7 reference queries covering adventure, culture, beaches, luxury, budget, family, and logistics travel styles.
- Both endpoints return the same RagSearchResponse structure: query, top_k, and results (destination_name, source_title, source_url, distance, chunk_preview, chunk_index).
- Added RagSearchRequest, RagChunkResult, RagSearchResponse, RagEvalCaseResponse, and RagEvalResponse Pydantic schemas.
- Evaluation cases score retrieval by checking if expected destinations appear in top-3 and top-5 results.
- Created app/routers/rag.py with both endpoints using existing RagService and dependency injection patterns.
- Added comprehensive tests for response shapes, destination filtering, pass/fail logic, and truncation.
- Added docs/rag_retrieval_tests.md explaining the endpoints, evaluation cases, and integration path to agent.

## RAG search input normalization

Summary:
- Added Pydantic v2 field_validator to RagSearchRequest.destination_filter.
- Normalizes common no-filter values ("none", "null", "all", "", whitespace-only) to None (case-insensitive).
- Strips whitespace from real destination names.
- Prevents users from accidentally passing "none" as a literal destination name in Swagger.
- Improves UX by accepting multiple ways to indicate "no filter".
